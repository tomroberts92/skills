# Standards brief

Tom's standards as a single rubric, used both for implementing and for reviewing. Paste it into every implementer and reviewer subagent prompt (subagents do not inherit tom-mode, so name the standards explicitly), and use it as the checklist for the `tom-review` skill. `python-best-practices` and the `principle-*` skills may auto-fire on a matching file, but tom-mode and this brief will not reach a subagent unless you include them.

- **Laziness and reuse.** Walk the YAGNI ladder before writing: does it need to exist, is it already in the codebase, in the standard library, a native feature, an existing dependency, one line, else the minimum that solves it. Do not add a dependency when stdlib, native, or existing code will do. See `principle-laziness-protocol` and `principle-subtract-before-you-add`.
- **Code.** `python-best-practices` (the smell table) for `.py`, `typescript-best-practices` for `.ts`/`.tsx`, `separating-pure-from-io`, and the principles `type-system-discipline`, `boundary-discipline`, `minimize-reader-load`, `foundational-thinking`.
- **Naming.** Precise and consistent names. Name the specific role, not a generic category (`data`, `manager`, `handler`, `helper` are warning signs); encode units and state (`timeoutMs`, `isLoaded`); booleans answer a yes/no question. If a precise, short name is hard to find, the entity is poorly defined, reshape it before naming. See `naming-things`.
- **Abstractions and failure modes.** Prefer deep modules to shallow pass-throughs, and flag a leaky abstraction whose interface exposes its implementation (the deletion test, see `improve-codebase-architecture`). Fail loudly, question every silent fallback and swallowed error. Specify exception handling, what is caught, what propagates, and what the caller sees; a bare `except` or a catch that quietly continues is a finding.
- **Edge cases and input domain.** For every changed function or component, walk its inputs against the edges, not just the happy path: null and undefined, empty collections, type coercion (`String(null)` is the truthy string `"null"`, `==` vs `===`), boundary and zero values, timezone and locale (`toISOString` is UTC, not the user's calendar day), async and lifecycle staleness (a value frozen at mount, a stale closure), and any guard removed in the diff. Frontend null, empty-state, and timezone edges are the usual silent-failure sources, an empty chart or a viewer that opens to a 404 rather than a thrown error.
- **Comments.** Readable code first. Comment where the code is complex or non-obvious, not the obvious. No in-body narration blocks, no stale or "mirrors legacy X" references, no ticket refs in comments.
- **Tests.** TDD, a failing test first for any behaviour change.
- **Verification bar.** Prove behaviour on the real artifact with data that exercises the new branch, not just green tests on existing seed data. See `principle-prove-it-works`.
